# 와이어프레임 — 6월 중간평가 · 피드백 보완 조치 (설계 ①②③)

> **근거:** `_workspace/00_input/requirements-midterm.md`(설계안), 운영계획 PPT 로드맵(6월=중간평가·점검·코칭, **등급·보상 미산정**), `domain-model.md`/`business-rules.md`, 시각언어 = `DESIGN.md`(Toss: Primary `#3182f6`, Dark `#191f28`, border-radius 0 사각, 컴팩트 고밀도, 글꼴 Pretendard).
> **범위:** ① mid_review 가드(등급·보상 차단 화면 처리) · ② 진척 점검 대시보드 · ③ 보완 조치(ActionItem) 추적 + 12월 패널 + `/eval` 허브 카피. **④(목표 재조정) 제외.**
> **원칙:** 기존 토큰·컴포넌트 **재사용**(전면 재작업 아님). 신규는 최소. 모든 신규 화면은 역할 인지(role-aware), 빈/로딩/에러 상태 명시, 대비 AA + 색+라벨 병기.
> **API 가정:** `_workspace/02_contract/contract-midterm.md`가 아직 없음 → 설계안 §2②③ 기준으로 데이터 shape을 가정하고 본 문서 §6에 명시. 계약 확정 시 필드명만 맞추면 됨.

## B. 공통 레이아웃 규약 (기존 그대로)

- 셸: `AppShell`(사이드바 216 + 헤더 52). 본문 = `PageContainer` > `PageHeader` + 카드 스택.
- 페이지 헤더: `PageHeader`(title 20px/700 `#191f28`, subtitle 13px `#6b7684`, 우측 슬롯 — 주기 셀렉터/액션). 중간평가 화면 헤더에는 `CycleTypeBadge`(중간/최종)와 주기 셀렉터 노출.
- 안내: 화면 상단 `InfoBanner`(tone `info`/`tip`/`warning`/`success`). 중간평가의 "비구속" 톤은 `tip`(앰버) 또는 `info`(블루).
- 카드: `Card`(흰 배경, 1px `#e5e8eb` 보더, radius 0, 헤더 15px/700 + 하단 보더). 그리드는 `lg`↑ 다열, `md`↓ 1열 스택.
- 상태: 로딩 = `Skeleton`(헤더+카드 골격), 에러 = `ErrorState`(재시도), 빈 = `EmptyState`, 권한 = `Forbidden`. 전부 `@/components/States` 재사용.

## D. 화면 RES — `/eval/result/[userId]` 의 **mid_review 분기** (블록 ①)

> 현행 결과 페이지는 `cycle.status`를 보지 않아 중간 시점에도 등급/보상 수치가 노출될 위험. mid_review에선 **수치 숨김 + 진척 요약**으로 대체.

```
┌─ /eval/result/[userId]  (cycle.status === 'mid_review') ──────────────┐
│ Breadcrumb / PageHeader (주기 셀렉터 — 중간 배지)                      │
├──────────────────────────────────────────────────────────────────────┤
│ InfoBanner (tip)  "중간 점검 결과예요 (비구속)"                        │
│   지금은 중간 점검 단계라 등급·보상은 아직 산정되지 않아요.            │
│   아래는 상반기 진척 요약이에요. 최종 등급은 12월 최종평가에서 확정돼요.│
├──────────────────────────────────────────────────────────────────────┤
│ ┌─ 다크 요약 카드 (재사용) ─ 단, 등급 박스 자리에 "중간 점검" 표시 ──┐  │
│ │  이름/소속        [종합: 점검중]  [성과중심: –]  [협업·성장: –]     │  │
│ │  하단: "등급·보상은 최종평가 완료 후 공개돼요."                     │  │
│ └─────────────────────────────────────────────────────────────────┘  │
│ ┌─ Card "상반기 진척 요약" ─ MidtermProgressTable (읽기전용) ────────┐  │
│ │   본인평가 진척 표 (= MT C-1 표) — 등급 컬럼 없이 달성률·신호까지만 │  │
│ └─────────────────────────────────────────────────────────────────┘  │
│ ┌─ Card "부서장 중간 피드백" (있으면) ─ CommentThread 읽기전용 ──────┐  │
│ └─────────────────────────────────────────────────────────────────┘  │
│ ※ 이의제기 버튼·ComparisonBar·종합점수·ScoreCard 전부 숨김             │
└──────────────────────────────────────────────────────────────────────┘
```

- **핵심:** `finalGrade`/`finalScore`/`byGroup` 점수·`percentile`·`companyAvg`·`ComparisonBar`·이의제기 CTA를 **렌더하지 않음**. 등급 박스는 "점검중"(`–`) 플레이스홀더.
- 백엔드가 mid_review에서 결과 집계 API를 400으로 막으면(설계안 ①), 프론트는 그 신호(또는 `cycle.status`)로 이 분기를 띄운다 — 수치 fetch 자체를 시도하지 않음(§6 가정 G3).

## E. 화면 FIN — `/eval/result/[userId]` 의 **calibration/closed 단계** 보완 조치 패널 (블록 ③ §2)

> 기존 최종 결과 화면(등급·비교·코멘트)은 그대로 두고, **맨 아래에 참고 패널 1개 추가**.

```
┌─ ... (기존 다크 요약 + 평가자 플로우 + 비교 + 코멘트) ... ────────────┐

### 진입점 결정 (요청 #3에 대한 제안)

**제안 = 신규 라우트 `/admin/midterm/rebaseline` 을 둔다.** 근거:
- `/admin/cycle`의 재오픈 사유 입력(`page.tsx:849`, placeholder "중간평가 목표 조정 반영")은 **단계 잠금/재오픈 토글**의 사유이지 KPI 단위 재조정이 아님 — 둘은 의미가 다르다(주기 잠금 사유 ≠ KPI별 목표 변경 사유). 한 모달에 욱여넣으면 "어떤 KPI를, 왜" 추적이 사라진다.
- ④는 **피평가자 선택 → KPI 표 편집 → 가중치 검증 → KPI별 사유**라 화면(테이블 + 이력 패널)이 필요. 모달로는 좁다.
- 다만 두 흐름은 **연결**한다: `/admin/cycle` 재오픈 사유 모달 본문에 보조 링크 "**KPI 목표를 조정하려면 → 목표 재조정 화면**"(`/admin/midterm/rebaseline`)을 추가. 잠금을 풀고(active 재오픈) 곧장 재조정으로 보내는 자연스러운 동선.
- `/eval/midterm`(②③)은 **본인·부서장의 점검/피드백** 화면이므로 거기에 두지 않는다(관리 액션과 분리). 단, 부서장은 `/admin/midterm/rebaseline`에 role 가드로 접근(자기 가시 범위 구성원만).

│   helper: "감사 로그·이력에 기록돼요. (필수)"                               │
└──────────────────────────────────────────────────────────────────────────┘

┌─ 저장 바 (sticky 하단) ───────────────────────────────────────────────────┐
│  좌: 변경 요약 "3개 KPI 변경 · 가중치 합 100% · 정성 30%"  (WeightSummaryBar)│
│  우: [되돌리기]  [조정 전 스냅샷 캡처 후 저장 ▸ primary]                    │
│  저장 비활성 조건(아래 ④-D 검증) 미충족 시 disabled + 사유 툴팁            │
└──────────────────────────────────────────────────────────────────────────┘
```

- **편집 가능 필드:** `targetValue`(정량) / `targetText`(정성) / `weight`. **편집 불가:** 과제명·category·group·measureType(KPI 정체성 — 변경 필요 시 별도 KPI 편집 흐름). 정성/정량 토글도 여기선 잠금.
- **변경 셀 강조:** 입력값이 원본과 다르면 셀에 `RebaselineChangedCell` 강조(좌 4px primary 바 + `primary-50` bg + 우상단 작은 점). 되돌리면 강조 해제. (component-spec ④ §1.)
- **행별 사유:** 그 행에 변경이 있으면 사유 칸이 활성+필수 표시(`*`), 미변경 행은 비활성·회색 "—". (전체 사유와 별개 — 백엔드가 행별을 지원하지 않으면 전체 사유만 필수, §④-G 가정.)

### ④-B-2. 가중치 합계 표시·검증 — WeightSummaryBar (신규, 요청 핵심)

`KPI 목표 재조정` 카드 우측 슬롯 + 저장 바에 동일 컴포넌트.

```
┌─ WeightSummaryBar ────────────────────────────────────────────────────────┐
│  가중치 합  100%  ✔        정성 KPI 합  30%  ✔                              │
│  └ 막대(0~100, 100 기준선) ────────────────█  성과중심 70 · 협업·성장 30    │
└───────────────────────────────────────────────────────────────────────────┘

  ▸ 합 ≠ 100:   가중치 합  93%  ⚠  → "합이 100%가 되어야 저장할 수 있어요 (현재 93%)"  (danger 인라인)
  ▸ 정성 > 30:  정성 KPI 합  35%  ⚠  → "정성 KPI 합이 30%를 넘었어요 (35%)"           (warning 인라인)
```

- **합 = 100** : 하드 검증(미충족 → 저장 disabled). 기존 `kpi-import/page.tsx:285`·`group-performance/page.tsx:175` 동일 정책.
- **정성 ≤ 30** : **소프트 권장**(초과 시 warning 경고, 저장은 차단하지 않음 — 기존 `kpi/page.tsx:867` "정성 비중은 권장, 제출 차단은 합 100만"과 일관). *(요청문은 "정성 ≤30"을 검증으로 적었으나, 기존 정책이 소프트이므로 일관성 위해 소프트 유지 — §④-G 가정 G④-3. 하드로 바꾸려면 플래그 한 줄.)*
- 실시간 합산은 **프론트 표시용**, 최종 검증은 백엔드(저장 시 합≠100이면 400).

### ④-B-3. 저장 흐름 & "스냅샷 캡처" 안내 (요청)

```
[조정 전 스냅샷 캡처 후 저장 ▸] 클릭
   → ConfirmDialog (Modal 재사용):
     ┌─ "재조정을 저장할까요?" ─────────────────────────────┐
     │ 변경 3개 KPI · 가중치 합 100%                          │
     │ 저장하면 조정 전 값이 "중간 조정 전" 스냅샷으로         │
     │ 보관되고, 변경 이력에 사유·변경자·시각이 기록돼요.       │
     │                          [취소]  [저장 ▸ primary]      │
     └──────────────────────────────────────────────────────┘
   → 성공: Toast(success) "재조정을 저장했어요. 조정 전 스냅샷이 캡처됐어요."
           + 표 강조 초기화(새 값이 '현재'가 됨) + 이력 패널 맨 위에 새 항목 추가(낙관적)
   → 실패(합≠100 등 400): Toast(danger) + 해당 검증 인라인 표시, 표 값 유지
```

- 저장은 **단일 트랜잭션 가정**: 변경된 KPI들을 1건의 PATCH로 일괄 전송 + 서버가 스냅샷 1개 캡처 + AuditLog 기록(§④-G A④-1).

## ④-C. 변경 이력 / diff 패널 — RebaselineHistory (신규 조립)

> 요청 #2. **기존 `KpiSnapshotDiff` UI 패턴 재사용** — `added/removed/changed` 구조 그대로. 재조정은 보통 `changed`(목표·가중치 변경)만 발생하므로 `changed` 행 중심.

```
┌─ Card "재조정 이력" ──── 우측: "{N}건" ──────────────────────────────────┐
│  (RB 화면 우측 패널 lg↑ / 하단 stacked md↓. 단계 무관 항상 조회 가능)      │
│ ┌─ RebaselineHistoryItem × N (최신순) ───────────────────────────────────┐│
│ │ ● 2026-06-08 14:20 · 김본부장(division_head)            [되돌릴 수 없음]││
│ │   사유: "상반기 시장 위축으로 매출 목표 하향, 수주 비중 상향"            ││
│ │   ┌─ diff (changed 행) ─────────────────────────────────────────────┐ ││
│ │   │ 매출액 달성   목표  120억 → 100억   가중치  40% → 35%             │ ││
│ │   │ 수주 건수     가중치  30% → 35%                                   │ ││
│ │   │ 협업 성과     목표  "분기 1회…" → "분기 2회…"                     │ ││
│ │   └─────────────────────────────────────────────────────────────────┘ ││
│ │   변경 없는 KPI 5개                                                     ││
│ ├────────────────────────────────────────────────────────────────────────┤│
│ │ ● 2026-06-05 …  (이전 재조정)                                          ││
│ └────────────────────────────────────────────────────────────────────────┘│
│  빈: EmptyState "아직 재조정 이력이 없어요."                               │
└──────────────────────────────────────────────────────────────────────────┘
```

- **diff 행 색(요청):** `before → after` 화살표 표기. **변경 = primary 계열**(화살표·after 값 `primary-600`), before 값은 `neutral-500` + 약한 취소선 톤(완전 line-through는 아님 — 가독성). 추가(added)=`success` 계열, 제거(removed)=`danger` 계열 — **등급색(grade-*)과 분리**. 색 정의는 component-spec ④ §3.
- **이력 항목 = 1회 재조정(=1 스냅샷).** `KpiSnapshotMeta`(label="중간 조정 전 …", createdAt, kpiCount) + `KpiSnapshotDiff`(changed/added/removed) 재사용. 변경자·사유는 AuditLog/스냅샷 메타에서(§④-G A④-2 — 기존 메타에 사유·변경자 없으면 백엔드 보강 필요, 가정 명시).
- **펼침/접힘:** 각 항목 기본 펼침(최신), 과거 항목 접힘 가능. diff가 길면 `changed` 우선, 미변경 수는 카운트만.
- 재조정은 되돌릴 수 없음(append-only 이력) — "되돌릴 수 없음" 칩(`neutral-100`). 잘못 저장 시 다시 재조정해 정정(새 이력 항목).

## ④-G. API 데이터 shape 가정 (contract-midterm.md 미기재 → 명시)

| # | 가정 | 근거/대체 |
|---|------|----------|
| A④-1 | `PATCH /api/v1/midterm/rebaseline` (일괄 1건): `{ cycleId, evaluateeId, reason, changes: [{ kpiId, targetValue?, targetText?, weight?, lineReason? }] }` → 서버가 **변경 KPI 일괄 적용 + "중간 조정 전" 스냅샷 1개 캡처 + AuditLog 기록**. 응답 `{ data: { snapshotId, changedCount } }`. **일괄 형태 채택**(원자성·스냅샷 1:1·이력 가독성). 단건 PATCH도 가능하나 스냅샷이 KPI별로 쪼개져 이력 추적이 흐려짐 → 일괄 권장. *백엔드가 단건만 지원하면 프론트가 변경 모아 순차 호출 + 묶음 스냅샷 요청으로 대체 가능.* |
| A④-2 | 이력 = 기존 `GET /cycles/:cycleId/kpi-snapshots?userId=` (`KpiSnapshotMeta[]`) + `GET …/:snapshotId/diff` (`KpiSnapshotDiff`) **재사용**. 단 메타에 **변경자(createdBy)·사유(reason)** 필드가 없으면 백엔드 보강 필요(`KpiSnapshotMeta`에 `reason?`, `createdByName?` 추가). label은 "중간 조정 전 · {YYYY-MM-DD}" 컨벤션. |
| A④-3 | 현재 확정 KPI 목록 = 기존 KPI 조회(`GET /cycles/:cycleId/kpis?userId=` 또는 동등) — `targetValue/targetText/weight/group/measureType/isQualitative` 포함. |
| A④-4 | 단계 게이팅: `PATCH …/rebaseline` 은 `cycle.status !== 'mid_review'` 면 **400**("중간 점검 단계에서만 목표를 재조정할 수 있어요."). ①의 게이팅 패턴과 대칭(다만 ①은 calibration/closed 허용, ④는 mid_review만). 프론트는 `cycle.status`로 선분기(저장 시도 회피). |
| A④-5 | 가중치 검증: 합 100 위반 시 백엔드 400. 정성 ≤30은 소프트(경고만, 저장 허용) — 기존 정책 일관(G④-3). |
| A④-6 | 권한: 생성/저장 = `hr_admin`·`division_head`·`team_lead` + 서비스단 "피평가자 상위장" 검증(ActionItem 생성과 동일 범위). 이력 조회 = 본인·부서장·HR. |

> 백엔드 ④ 계약 확정 시 위 경로/필드명을 1:1로 맞추면 됨. shape 기준은 위 표 + component-spec ④ §0 타입.
