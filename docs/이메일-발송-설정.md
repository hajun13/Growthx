# 이메일 발송 설정 (SMTP)

대상 코드: `apps/api/src/modules/notifications/mail.service.ts`, `notifications.service.ts`, `apps/api/src/modules/midterm/midterm-notify.service.ts`, `midterm-review-flow.service.ts`. 배포 설정: `docker-compose.yml`(로컬/개발), `docker-compose.prod.yml`(맥, `hr.energyx.co.kr`, `docs/deploy-kpi-macos.md` 참고), `.env.example`.

## 1. 시작 전에 메일 서버 담당자에게 받아야 할 값

| 값 | 용도 | 전형적인 값 |
|---|---|---|
| SMTP 호스트 | `mail.service.ts` 가 `nodemailer.createTransport({ host, port, ... })` 에 그대로 넘기는 접속 주소 | `smtp.energyx.co.kr` 같은 사내 릴레이, 또는 Microsoft 365 등 외부 서비스 주소 |
| 포트 | 마찬가지로 `createTransport` 에 그대로 전달 | STARTTLS면 `587`, 암묵적 TLS면 `465`, 사내 인증 없는 릴레이는 `25` 인 경우도 있음 |
| TLS 방식 | 코드는 `secure: process.env.SMTP_SECURE === 'true'` 로만 판단 — 포트가 465(암묵적 TLS)면 `true`, 587(STARTTLS)이면 `false` 로 넣어야 함. 포트와 이 값의 조합이 틀리면 접속 자체가 실패함 | 465↔true, 587↔false |
| 인증 계정 여부 | 코드는 `auth: user && pass ? { user, pass } : undefined` — **아이디·비밀번호를 둘 다 주지 않으면 인증 정보를 아예 보내지 않음**. 메일 서버가 인증 없는 릴레이(내부망 IP 허용 등)를 허용하는지, 아니면 반드시 인증 계정이 있어야 하는지 확인 필요 | 인증 필요 시 발신 전용 계정 하나 발급받기 |
| 발신 주소 결정 | `SMTP_FROM` — 미설정 시 코드 기본값 `no-reply@energyx.co.kr` 로 발송됨(`mail.service.ts:39`). **no-reply 사서함으로 할지, 실제 사람이 답장을 받는 사서함으로 할지 결정 필요** — 이의제기 답변·중간점검 반려처럼 "확인 후 되물을 수도 있는" 메일이 섞여 있어 no-reply로 굳히면 회신 경로가 없어짐을 감안 | `no-reply@energyx.co.kr` 또는 `hr@energyx.co.kr` |
| 발신 주소 인증 요구 여부 | 실제 발송 단계에서 자주 걸리는 지점: 많은 메일 서버가 "발신 주소(From)가 인증 계정과 다르면 거부"하는 정책을 씀. **`SMTP_FROM` 으로 정한 주소가 `SMTP_USER` 인증 계정과 같아야 하는지, 아니면 별칭(alias)으로 허용되는지 메일 서버 담당자에게 확인** | — |
| 서비스 접속 주소 | `APP_BASE_URL` — 메일 본문 링크에 붙는 절대 URL(§6 참고) | `https://hr.energyx.co.kr` |

이 7가지 중 앞의 5개(호스트·포트·TLS·인증 계정·발신 주소 정책)는 메일 서버를 운영하는 쪽에 직접 물어봐야 하는 항목입니다. 코드는 이 값들이 무엇이어야 하는지 전혀 검증하지 않고 그대로 nodemailer 에 전달하기 때문에, 틀린 값이 들어가면 "설정은 됐는데 안 나가는" 상태가 됩니다(§7 트러블슈팅 참고).

## 2. 값이 들어가는 곳 — 호스트 `.env` 와 컨테이너 환경의 관계

`MailService` 는 `process.env.SMTP_HOST` 등을 **컨테이너 안에서** 직접 읽습니다(`mail.service.ts:16-19`). 즉 값이 실제로 반영되려면 두 단계를 모두 거쳐야 합니다.

1. **호스트 `.env` 파일**(로컬/개발은 리포 루트 `.env`, 운영은 `docs/deploy-kpi-macos.md` 기준 `.env.prod`)에 `SMTP_HOST`, `SMTP_PORT`, `SMTP_SECURE`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM`, `APP_BASE_URL` 키를 채웁니다. `.env.example` 에 주석 처리된 자리가 이미 있습니다(41~48행).
2. `docker-compose.yml`(57~67행)과 `docker-compose.prod.yml`(57~66행)의 `api` 서비스 `environment:` 블록이 이 값들을 **컨테이너 환경변수로 다시 전달**합니다. 두 파일 모두 이미 `SMTP_HOST: ${SMTP_HOST:-}` 형태로 통과시키도록 배선되어 있으니, 이 문서 작성 시점에는 **`.env`/`.env.prod` 를 채우는 것 외에 compose 파일을 고칠 필요는 없습니다.**

이 관계를 이해해 두면 문제가 생겼을 때 어디를 봐야 할지 스스로 판단할 수 있습니다 — docker-compose 파일 자체에 있는 주석이 이 함정을 정확히 짚고 있습니다: "SMTP_* 를 여기서 통과시키지 않으면 호스트 `.env` 를 채워도 컨테이너 안에서는 영영 콘솔 폴백으로만 동작한다." 즉 **`.env` 파일에 값이 있는 것과, 컨테이너 프로세스가 그 값을 실제로 보는 것은 별개**이고, 지금은 compose 쪽 배선이 이미 되어 있으니 `.env` 를 채우는 쪽만 신경 쓰면 됩니다. 반대로 나중에 새로운 환경변수를 추가한다면, `.env.example` 뿐 아니라 **두 compose 파일의 `environment:` 블록에도** 같은 키를 추가해야 컨테이너가 실제로 읽습니다.

값을 바꾼 뒤에는 컨테이너를 **재생성**해야 합니다. `MailService` 의 nodemailer transporter 는 컨테이너 프로세스 기동 시(생성자에서) 한 번만 만들어지므로, 단순 `docker compose restart api` 는 안전하지 않을 수 있습니다(같은 컨테이너를 그대로 재시작하면 반영되긴 하지만, 값이 바뀐 걸 compose 가 감지해 재생성하도록 아래 명령을 권장합니다):

```
docker compose --env-file .env -f docker-compose.yml up -d api
# 운영이면
docker compose --env-file .env.prod -f docker-compose.prod.yml up -d api
```

## 3. 검증 — 확신을 점점 키워가는 순서

### 3-1. 지금(SMTP 미설정) 콘솔 폴백이 실제로 동작하는지 확인

`SMTP_HOST`/`SMTP_PORT` 가 비어 있으면 `mail.service.ts:20` 의 `this.enabled = Boolean(host && port)` 가 `false` 가 되어, 발송 시도마다 아래 형식으로 로그만 남기고 끝납니다(`mail.service.ts:34`):

```
[MAIL:dev-fallback] to=<수신자 이메일> subject="<제목>" body="<본문>"
```

(NestJS Logger 를 통해 나가므로 실제 stdout에는 `[Nest] <pid>  - <timestamp>   LOG [MailService] ` 가 앞에 붙습니다.)

확인 방법:

```
docker compose logs -f api
```

를 띄워 둔 채로, 알림을 발생시키는 아무 동작(예: HR 계정으로 KPI 반려, 또는 아래 3-3의 중간점검 흐름)을 하나 실행하면 위 로그 줄이 찍혀야 합니다. **이 상태가 지금(2026-07-23 기준) 리포지토리의 기본값입니다** — 루트 `.env` 를 확인해 보면 `SMTP_*`/`APP_BASE_URL` 키 자체가 아직 없습니다.

### 3-2. 실제 발송 확인

`.env`/`.env.prod` 를 채우고 §2의 명령으로 `api` 컨테이너를 재생성한 뒤, 같은 방식으로 알림을 하나 발생시켜 봅니다.

- **성공했을 때 코드가 로그를 남기지 않는다는 점을 미리 알아두세요.** `mail.service.ts:44` 는 발송 성공 시 `{ sent: true, mode: 'smtp' }` 를 반환만 하고 별도 로그를 찍지 않습니다. 즉 "성공 로그"는 존재하지 않고, **"콘솔 폴백 로그(`[MAIL:dev-fallback]`)가 더 이상 찍히지 않는 것"이 성공의 첫 신호**입니다.
- 실패하면 `mail.service.ts:46` 이 남기는 다음 형식의 경고가 찍힙니다: `mail send failed to <수신자>: <에러 메시지>`. 에러 메시지는 nodemailer 가 던진 그대로이므로 인증 실패·접속 거부 등 원인이 대체로 드러납니다.
- 어느 쪽이든 **실제 수신함으로 최종 확인**하세요 — 로그가 조용하다고 반드시 도착했다는 뜻은 아니고(예: 서버가 접수 후 조용히 버리는 경우), 로그는 "우리 서버가 넘겼는지"만 알려줍니다.
- 참고: 인앱 알림(알림센터)은 메일 성공 여부와 무관하게 항상 생성됩니다(`notifications.service.ts:84-91` 이 메일 발송과 별개로 DB에 저장). 사용자가 "메일은 못 받았는데 알림센터엔 떠 있다"고 하면 정상 동작입니다.

### 3-3. 중간점검(mid-term) 플로우 종단 확인

중간점검 알림은 상태 전이가 커밋된 뒤 `MidtermNotifyService.dispatch()` 가 소비합니다. 실제 화면(`/eval/midterm`)에서 아래 순서를 밟으며 매 단계 로그와 수신함을 확인하세요.

1. HR 계정으로 중간점검 개시(`중간점검이 개시되었습니다`) → 1차 평가자에게 메일이 가야 함.
2. 1차 평가자가 코멘트 등록(`중간점검 코멘트가 등록되었습니다`) → 피평가자 본인에게.
3. 피평가자가 수정본 제출(`중간점검 수정본이 제출되었습니다`) → 최종(2차) 평가자에게.
4. 최종 평가자가 승인(`중간점검이 마무리되었습니다`) → 피평가자 **와 1차 평가자 둘 다**에게. 반려(`중간점검이 반려되었습니다`)면 피평가자에게만.

`APP_BASE_URL` 을 아직 안 채운 상태로 1번을 실행하면, `midterm-notify.service.ts:24-29` 가 **딱 한 번만** 아래 경고를 남깁니다(수신자마다 반복하면 로그가 묻히기 때문에 프로세스당 1회 제한):

```
APP_BASE_URL 이 설정되지 않아 중간점검 알림 링크가 상대경로로 나갑니다(메일에서 클릭 불가). 환경변수를 설정해 주세요.
```

이 경고가 뜨면 §6 대로 `APP_BASE_URL` 을 채우고 다시 검증하세요. 다수 수신자 중 일부만 발송에 실패하면(`midterm-notify.service.ts:41-43`) `midterm notify failed (<type>, userId=<id>): <에러 메시지>` 형식으로 실패한 사람의 userId 까지 로그에 남습니다 — 알림 실패가 업무 자체(상태 전이)를 막지는 않는 설계입니다.

## 4. 이 시스템이 보내는 메일 — 중간점검 플로우

`midterm-review-flow.service.ts` 가 만드는 `NotifyIntent` 를 기준으로 정리했습니다. 제목은 `notifications.service.ts` 의 `subjectForType()` 이 실제로 만들어내는 값이고, 모든 제목 앞에 `[에너지엑스 인사 평가] ` 가 붙습니다.

| 알림 타입 | 수신자 | 트리거 | 이메일 제목 |
|---|---|---|---|
| `midterm_opened` | 1차 평가자 (중복 제거) | HR 이 중간점검 개시(`open()`, dryRun 아님) | 중간점검이 개시되었습니다 |
| `midterm_comment_received` | 피평가자 본인 | 1차 평가자가 코멘트 등록(`comment()`, pending→commented) | 중간점검 코멘트가 등록되었습니다 |
| `midterm_revision_submitted` | 최종(2차) 평가자 (있을 때만) | 피평가자가 수정본 제출(`submitRevision()`, commented/returned→revised) | 중간점검 수정본이 제출되었습니다 |
| `midterm_closed` | 피평가자 **+ 1차 평가자** | 최종 평가자 승인(`decide()` → closed) | 중간점검이 마무리되었습니다 |
| `midterm_returned` | 피평가자만 | 최종 평가자 반려(`decide()` → returned) | 중간점검이 반려되었습니다 |

추가로 알아둘 것: **HR 되돌림(`reopen()`)과 재배정(`reassign()`)은 메일을 전혀 보내지 않습니다** — 코드가 각각 `notify: []` 를 명시적으로 반환합니다(`midterm-review-flow.service.ts:655`, `:704`). 재배정으로 평가자가 바뀌었다는 사실을 당사자에게 알리고 싶다면 별도 채널(사내 공지 등)이 필요합니다.

### SMTP를 켜면 같이 나가는 다른 메일들 (사거리)

SMTP는 애플리케이션 전체가 공유하는 하나의 스위치입니다. 켜는 순간 중간점검 메일만이 아니라 아래도 전부 실제로 나갑니다.

- `notifications.service.ts` 의 **일정 마감 리마인더**(`deadline_d{N}`) — 매일 09:00 크론(`reminder.scheduler.ts`)이 자동 발송하거나, HR 이 `POST /notifications/run-reminders` 로 수동 발송.
- `kpis.service.ts` — KPI 반려(`kpi_rejected`), 결재 확정(`kpi_confirmed`), 다음 결재자 승인 대기(`kpi_approval_pending`).
- `evaluations.service.ts` — 결과 확정(`result_finalized`), 수정요청(`evaluation_revision_requested`), 반려(`evaluation_rejected`).
- `appeals.service.ts` — 이의제기 답변(`appeal_answered`), 최종 결정(`appeal_decided`).

**발견한 불일치 — 제목이 안 붙는 알림 타입이 있습니다.** `notifications.service.ts:270-288` 의 `subjectForType()` 매핑에는 `kpi_confirmed`, `kpi_approval_pending`, `evaluation_revision_requested`, `evaluation_rejected` 4개 타입이 등록돼 있지 않습니다. 이 타입들은 전부 코드 안에서 실제로 발송되는데(`kpis.service.ts:335,343`, `evaluations.service.ts:771,813`), 매핑에 없으면 함수가 기본값 `'새로운 알림'` 을 반환하므로 **이메일 제목이 전부 `[에너지엑스 인사 평가] 새로운 알림` 으로 뭉개집니다.** (참고로 `kpi_confirmed`/`kpi_approval_pending` 은 `apps/web/lib/ui.ts` 의 인앱 알림센터에는 `type.startsWith('kpi')` 규칙 덕에 정상 분류되지만, `evaluation_*` 2종은 인앱 쪽 `notificationStyle`/`notificationCategory`/`notificationNavKey` 어디에도 등록돼 있지 않아 인앱에서도 제목 없이 "새 알림이 있어요"로만 뜹니다.) 중간점검 5종은 인앱·이메일 양쪽에 전부 정확히 등록돼 있어 이 문제가 없습니다. SMTP를 켜기 전에 고칠지, 우선 그대로 켤지는 제품 판단이 필요합니다 — 이 문서는 발견 사실만 남깁니다.

## 5. 메일 본문에 코멘트를 넣지 않는다는 설계 — 그리고 이미 어긴 곳

`midterm-notify.service.ts` 상단 주석이 원칙을 명시합니다: "본문은 '요약 + 링크'만 — 평가 코멘트 전문은 메일에 싣지 않는다." 실제로 `dispatch()` 는 코멘트 원문을 절대 메일에 넣지 않고, `중간점검 코멘트가 등록됐어요. 조정 검토 요청 KPI ${adjustCount}건` 같은 **생성된 요약 문구 + 앱 링크**만 보냅니다(`midterm-review-flow.service.ts` 의 `comment()`/`submitRevision()`/`decide()` 가 만드는 `message` 참고).

이걸 지키는 이유: 평가 코멘트·반려 사유는 조직 내부 피드백이라, 메일 본문에 그대로 실으면 그 순간부터 회사 메일함·백업·검색 인덱스에 원문이 영구히 남습니다. 앱 안에서만 노출하면 RBAC(누가 볼 수 있는지)를 앱이 계속 통제할 수 있지만, 메일로 나가는 순간 그 통제를 벗어납니다. **나중에 "링크 대신 내용을 바로 보여주면 편하지 않을까" 하는 요청이 와도, 이 설계를 되돌리는 것이므로 신중해야 합니다.**

**다만 이미 이 원칙을 어기고 있는 곳이 있습니다.** 아래 3개 알림은 자유서술 사유(`dto.reason`)를 메일 본문에 그대로 넣고 있습니다:

- `kpis.service.ts:401` — `kpi_rejected`: `` `KPI "${kpi.title}"가 반려되었어요: ${dto.reason}` ``
- `evaluations.service.ts:775` — `evaluation_revision_requested`: `` `평가에 수정요청이 등록되었어요: ${dto.reason}` ``
- `evaluations.service.ts:817` — `evaluation_rejected`: `` `평가가 반려되었어요: ${dto.reason}` ``

즉 검토자가 입력한 반려·수정요청 사유 원문이 SMTP를 켜는 순간 그대로 수신자의 회사 메일함에 남습니다. 중간점검 모듈만 설계를 지키고 있고, KPI/평가 반려 경로는 아직 지키지 않는 상태입니다 — SMTP를 켜기 전에 제품 차원에서 이 3곳을 그대로 둘지(반려 사유 정도는 괜찮다고 판단) 아니면 중간점검과 동일하게 요약+링크로 바꿀지 결정이 필요합니다.

## 6. `APP_BASE_URL`

`midterm-notify.service.ts:20,30` 이 이 값을 읽어 `${baseUrl}/eval/midterm` 링크를 만듭니다. 미설정이면 `baseUrl` 이 빈 문자열이 되어 링크가 `/eval/midterm` 같은 **상대경로**로 나갑니다 — 상대경로는 메일 클라이언트(웹메일·아웃룩 등)에서 어느 도메인 기준인지 알 수 없어 **클릭이 되지 않거나 엉뚱한 곳으로 열립니다.** (동작 자체는 막지 않습니다 — 알림이 아예 안 나가는 것보다는 낫다는 판단으로, 첫 발송 때 경고 로그만 한 번 남기고 계속 진행합니다.)

이 배포에서 넣어야 할 값은 실제 서비스 접속 주소입니다. `docker-compose.prod.yml` 주석과 `docs/deploy-kpi-macos.md` 를 보면 운영 도메인은 `hr.energyx.co.kr` 이므로:

```
APP_BASE_URL=https://hr.energyx.co.kr
```

로컬/개발 스택이면 `http://localhost:3000` (또는 `WEB_PORT` 로 바꾼 포트).

## 7. 트러블슈팅

| 증상 | 어디를 볼지 | 유력한 원인 |
|---|---|---|
| 인증 거부(콘솔에 `mail send failed to ... : ...Authentication...` 류) | `docker compose logs api` | `SMTP_USER`/`SMTP_PASS` 오타, 또는 둘 중 하나만 채워서 코드가 인증 정보를 아예 안 보냄(`mail.service.ts:26` — `user && pass` 둘 다 있어야 인증 시도) |
| 발신 주소 거부(relay denied, sender rejected 류) | 같은 로그 | `SMTP_FROM` 이 메일 서버가 허용하지 않는 주소 — 인증 계정과 다른 주소를 From 으로 쓰는 걸 막는 서버가 많음(§1) |
| TLS/포트 불일치(handshake 실패, ECONNRESET, timeout) | 같은 로그 | `SMTP_PORT`=587인데 `SMTP_SECURE=true`(또는 그 반대)로 잘못 짝지음, 방화벽이 해당 포트를 막고 있음 |
| 로그도 조용하고 여전히 콘솔 폴백만 찍힘 | `docker compose exec api env \| grep SMTP` 로 컨테이너 안 실제 값 확인 | `SMTP_HOST`/`SMTP_PORT` 둘 다 있어야 `enabled=true`(`mail.service.ts:20`) — 호스트만 채우고 포트를 비워두면 여전히 콘솔 폴백. 또는 §2의 컨테이너 재생성을 안 해서 이전 값(transporter)이 메모리에 남아있음 |
| "발송됐다"는데 도착 안 함 | 메일 서버 쪽 큐/스팸함, 수신자 스팸함 | §3-2에서 설명했듯 성공은 로그를 남기지 않음 — nodemailer 가 서버에 접수시킨 것과 실제 도착은 별개. SPF/DKIM 미설정으로 스팸 처리됐을 가능성도 메일 서버 담당자에게 확인 |
| 메일은 왔는데 링크가 클릭 안 됨/이상한 주소로 감 | 메일 본문의 링크 문자열 | `APP_BASE_URL` 미설정(§6) — 로그에 1회성 경고가 남았는지 확인 |
| 일부 사람만 못 받음(다른 사람은 받음) | `midterm notify failed (...), userId=...` 로그 | 그 userId 사용자의 `email` 필드가 비어있거나 잘못됨(`notifications.service.ts:93-96` 이 email 없으면 조용히 건너뜀 — 이 경우는 아예 에러 로그도 없음) |
