# 이메일(SMTP) 발송 — 맥 운영 서버 작업 지시서

> 이 문서는 **맥(운영, `hr.energyx.co.kr`)에서 SMTP를 실제로 켜는 작업**의 지시서입니다.
> 코드 동작·알림 종류·트러블슈팅 전반은 `docs/이메일-발송-설정.md` 를 함께 보세요. 이 문서는
> **"Direct Send 가 막혔다"는 확정 이후 남은 경로와, 맥에서 밟을 순서**만 다룹니다.
>
> 작성 시점: 2026-07-23 / 대상 브랜치: `feat/midterm-two-stage`

## 0. 결론부터

- **Direct Send(`energyx-co-kr.mail.protection.outlook.com:25`)는 사용 불가로 확정**되었습니다(테넌트에서 허용 안 됨).
- 따라서 **계정 인증 기반 경로**(아래 §2의 B 또는 C)로 가야 하며, **IT/테넌트 관리자로부터 값을 받기 전까지는 맥에서 할 수 있는 일이 없습니다.**
- 값이 오기 전에 **먼저 끝낼 수 있는 준비 작업**(§4 안전장치, §5 미결 2건)이 있으니 그건 병행하세요.

## 1. 확정된 사실 (재조사 불필요)

### 회사 메일 인프라
| 항목 | 값 |
|---|---|
| MX | `energyx-co-kr.mail.protection.outlook.com` → **Microsoft 365 (Exchange Online)** |
| SPF | `v=spf1 include:spf.protection.outlook.com include:mail.stibee.com -all` |

SPF가 **`-all` 하드페일**입니다. 즉 M365를 거치지 않고 임의 서버에서 직접 보내면 자기 회사 메일함에서도 스팸/거부 처리됩니다. → 반드시 M365를 통해 나가야 합니다.

### 리포지토리 상태
| 항목 | 상태 |
|---|---|
| 루트 `.env` | `SMTP_*`·`APP_BASE_URL` 키 **없음** → 콘솔 폴백 |
| `.env.prod.example` | **SMTP 블록 자체가 없음** → 맥에서 `.env.prod` 만들 때 수동 추가 필요 (§6) |
| `docker-compose.prod.yml` (57~66행) | `SMTP_HOST/PORT/SECURE/USER/PASS/FROM`·`APP_BASE_URL` **통과 배선 완료** — compose 수정 불필요 |
| `docker-compose.yml` (57~67행) | 동일 |

### 코드 동작 (`apps/api/src/modules/notifications/mail.service.ts`)
- `enabled = Boolean(host && port)` — **호스트·포트 둘 다** 있어야 실제 발송. 하나만 채우면 계속 콘솔 폴백.
- `auth: user && pass ? {...} : undefined` — **아이디·비밀번호 둘 다** 있어야 인증 시도. 하나만 채우면 인증 없이 접속해 거부됨.
- `secure: SMTP_SECURE === 'true'` — 465면 `true`, 587/25면 `false`. 포트와 짝이 틀리면 접속 자체 실패.
- **발송 성공 시 로그를 남기지 않음.** `[MAIL:dev-fallback]` 로그가 멈추는 것이 성공의 첫 신호.
- 실패 시: `mail send failed to <주소>: <원인>` (warn)

### 맥 환경 제약 (경로 선택에 영향)
- 맥은 `hr` CNAME → `xxxx.iptime.org` DDNS 뒤에 있음(`docs/deploy-kpi-macos.md:30`) → **공인 IP가 유동일 가능성 높음.**
  → IP 기반 경로(**커넥터 릴레이**, SPF에 `ip4:` 추가)는 IP가 바뀌는 순간 조용히 죽습니다. **부적합.**
- 국내 ISP는 **아웃바운드 25번 포트를 차단**하는 경우가 흔합니다. → 587 기반 경로가 안전합니다.
- 위 두 이유로 **계정 인증 + 587** 조합(B 또는 C)이 이 환경에 맞습니다.

## 2. 남은 경로 2개

### B. HVE (High Volume Email) — 가능하면 1순위
```
SMTP_HOST=smtp-hve.office365.com
SMTP_PORT=587
SMTP_SECURE=false
SMTP_USER=<HVE 계정>
SMTP_PASS=<HVE 계정 비밀번호>
```
- 사내 대량 알림용으로 마이크로소프트가 별도로 제공하는 경로. **사서함 라이선스를 소모하지 않음.**
- 기본적으로 **자기 테넌트 내부 수신자 대상**. 이 앱은 수신자가 전부 `@energyx.co.kr` 임직원이므로 요건에 맞습니다.
- 관리자가 Exchange 관리센터에서 HVE 계정을 생성해야 합니다.
- ⚠️ 테넌트 플랜·지역·시점에 따라 제공 여부와 인증 방식(기본인증 허용 여부)이 다를 수 있으니 **관리자에게 현재 가능한지 확인**이 필요합니다.

### C. 일반 사서함 + SMTP AUTH — 현실적 대안
```
SMTP_HOST=smtp.office365.com
SMTP_PORT=587
SMTP_SECURE=false
SMTP_USER=no-reply@energyx.co.kr
SMTP_PASS=<앱 비밀번호 또는 계정 비밀번호>
```
- 가장 익숙하지만 M365에서 **막혀 있는 게 기본값**입니다. 관리자가 다음을 해줘야 합니다:
  1. `no-reply@energyx.co.kr` 사서함 생성(라이선스 1개 소모)
  2. **해당 사서함에 SMTP AUTH 활성화**(테넌트 전역이 꺼져 있으면 전역 예외도 필요)
  3. 보안 기본값/조건부 액세스가 기본인증을 막고 있으면 **이 계정에 대한 예외** 또는 앱 비밀번호 발급
- 계정 비밀번호가 만료·변경되면 발송이 멈추므로, **만료 정책 제외**를 함께 요청하세요.

> ❌ **커넥터 릴레이(IP 기반)** 는 위 §1의 유동 IP 문제로 제외합니다.
> ❌ **Direct Send** 는 테넌트에서 차단 확정.

## 3. IT/테넌트 관리자에게 보낼 요청문 (복붙용)

> 사내 인사평가 시스템(`hr.energyx.co.kr`, 맥 서버)에서 **임직원에게 알림 메일**을 발송하려 합니다.
> 수신자는 전원 `@energyx.co.kr` 내부 사용자이고, 외부 발송은 없습니다.
> Direct Send 는 사용 불가로 확인되어, 아래 둘 중 하나를 요청드립니다.
>
> **(1순위) HVE(High Volume Email) 계정 발급**
> - 접속: `smtp-hve.office365.com:587` (STARTTLS)
> - 필요한 것: HVE 계정 ID / 비밀번호
>
> **(2순위) `no-reply@energyx.co.kr` 사서함 + SMTP AUTH 활성화**
> - 접속: `smtp.office365.com:587` (STARTTLS)
> - 필요한 것: 계정 ID / 앱 비밀번호, 해당 사서함 SMTP AUTH 활성화, 비밀번호 만료 정책 제외
>
> 추가 확인 부탁:
> - 발신 주소(From)를 `no-reply@energyx.co.kr` 로 쓸 때, **인증 계정과 다른 주소**를 From 으로 허용하는지 (별칭 허용 여부)
> - 발송 서버는 유동 IP(DDNS) 환경이라 **IP 기반 커넥터는 부적합**합니다. 계정 인증 방식으로 부탁드립니다.

## 4. 값이 오기 전에 해둘 것 — 안전장치 (권장, 코드 변경)

SMTP는 **앱 전역 스위치 하나**입니다. 켜는 순간 중간점검 알림뿐 아니라 **매일 09:00 크론**
(`apps/api/src/modules/notifications/reminder.scheduler.ts`)의 마감 리마인더가 **전 사용자(실명부 117명)에게 실제로 발송**됩니다.

현재 코드에는 **테스트 발송 수단도, 수신자 제한 수단도 없습니다.** 켜기 전에 아래 2개를 넣는 것을 권합니다.

1. **수신자 allowlist / 리다이렉트** — `MAIL_ALLOWLIST` 같은 env 를 `MailService.send()` 앞단에 두어, 초기에는 실제 발송을 **작업자 본인 주소로만** 제한. 전체 오픈은 검증 후.
   - 넣는다면 `docker-compose.yml`·`docker-compose.prod.yml` 의 `environment:` 블록에도 **같은 키를 추가**해야 컨테이너가 읽습니다(§1의 통과 배선 규칙).
2. **테스트 발송 경로** — 지금은 KPI 반려 같은 **실제 업무 동작**을 해야만 메일이 나갑니다. HR 전용 `POST /notifications/test-mail`(수신자 지정) 하나면 검증이 훨씬 안전해집니다.
   - 급하면 §7의 컨테이너 1회용 스크립트로 대체 가능합니다.

## 5. 켜기 전 제품 결정 2건 (미결)

`docs/이메일-발송-설정.md` §4·§5에서 발견된 사항. **SMTP를 켜는 순간 되돌릴 수 없습니다.**

1. **제목이 `새로운 알림` 으로 뭉개지는 4개 타입** — `notifications.service.ts` 의 `subjectForType()` 에 `kpi_confirmed`, `kpi_approval_pending`, `evaluation_revision_requested`, `evaluation_rejected` 가 등록돼 있지 않음. 켜기 전에 매핑 추가 권장(간단).
2. **반려·수정요청 사유 원문이 메일 본문에 그대로 실림** — `kpis.service.ts:401`, `evaluations.service.ts:775`, `:817`. 검토자가 쓴 자유서술 사유가 수신자의 **회사 메일함·백업·검색 인덱스에 영구 저장**됩니다. 중간점검 모듈은 "요약+링크" 원칙을 지키는데 이 3곳만 어기고 있습니다.
   → 그대로 둘지, 중간점검처럼 요약+링크로 바꿀지 **결정 필요**.

## 6. 맥에서의 작업 순서

### 6-1. `.env.prod` 에 SMTP 블록 추가
`.env.prod.example` 에는 SMTP 키가 없으므로 **직접 추가**합니다. (예: 경로 B/HVE)

```
# ── 메일(SMTP) ── M365. Direct Send 불가 → 계정 인증 경로.
SMTP_HOST=smtp-hve.office365.com
SMTP_PORT=587
SMTP_SECURE=false
SMTP_USER=<발급받은 계정>
SMTP_PASS=<발급받은 비밀번호>
SMTP_FROM=no-reply@energyx.co.kr
# 메일 본문 링크의 절대 주소. 없으면 상대경로가 되어 메일에서 클릭 불가.
APP_BASE_URL=https://hr.energyx.co.kr
```

경로 C면 `SMTP_HOST=smtp.office365.com`, `SMTP_USER=no-reply@energyx.co.kr`.

> `.env.prod` 는 **커밋 금지**(`.gitignore` 로 차단됨). 같은 키를 `.env.prod.example` 에는 **빈 값으로** 추가해 두면 다음 배포자가 빠뜨리지 않습니다.

### 6-2. 접속만 먼저 검증 (앱 건드리기 전)
컨테이너에 nodemailer 가 이미 있으므로 단독 확인이 가능합니다. **여기서 통과하면 값이 맞다는 게 증명**되고, 실패하면 원인이 앱이 아니라 메일 서버·계정 쪽임이 확정됩니다.

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml up -d api

docker compose --env-file .env.prod -f docker-compose.prod.yml exec api \
  node -e "const n=require('nodemailer');const t=n.createTransport({host:process.env.SMTP_HOST,port:+process.env.SMTP_PORT,secure:process.env.SMTP_SECURE==='true',auth:process.env.SMTP_USER&&process.env.SMTP_PASS?{user:process.env.SMTP_USER,pass:process.env.SMTP_PASS}:undefined});t.verify().then(()=>console.log('CONNECT OK')).catch(e=>console.log('FAIL',e.message))"
```

### 6-3. 컨테이너가 값을 실제로 보는지 확인
값 변경 후에는 **`restart` 가 아니라 `up -d`** 로 재생성합니다(transporter 는 기동 시 1회 생성).

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml exec api env | grep -E "SMTP|APP_BASE_URL"
```

### 6-4. 실제 1통 발송
allowlist(§4)를 넣었다면 본인 주소로. 없다면 **본인 계정 대상 동작 하나**만 유발하세요.

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml logs -f api
```
- `[MAIL:dev-fallback]` 이 **더 이상 안 찍히면** 성공 신호(성공 로그는 없음).
- `mail send failed to ...` 가 뜨면 §7.
- **최종 판단은 수신함으로.** 스팸함도 확인.

### 6-5. 중간점검 5단계 종단 확인
`docs/이메일-발송-설정.md` §3-3 순서대로. 메일 링크가 **클릭되는지**(=`APP_BASE_URL` 반영) 함께 확인.

### 6-6. 전체 오픈 전 마지막 관문
다음날 09:00 크론 로그로 **발송 규모를 먼저 확인**한 뒤 allowlist 를 해제하세요.
```
마감 리마인더 자동 발송 — N건 단계 / M명
```

## 7. 트러블슈팅 (이 경로에서 자주 나오는 것)

| 증상 | 유력한 원인 |
|---|---|
| `CONNECT OK` 인데 발송만 실패 | From 주소 거부 — `SMTP_FROM` 이 인증 계정과 다른데 별칭이 허용 안 됨(§3 마지막 질문) |
| 인증 거부(`Authentication unsuccessful`) | 사서함에 SMTP AUTH 미활성, 보안 기본값/조건부 액세스가 기본인증 차단, 또는 `SMTP_USER`/`SMTP_PASS` 중 하나만 채움 |
| timeout / ECONNRESET | 587 아웃바운드 차단(방화벽·ISP), 또는 `SMTP_SECURE` 와 포트 짝 오류 |
| 여전히 콘솔 폴백만 | `SMTP_HOST`·`SMTP_PORT` 중 하나 누락, 또는 `up -d` 재생성 안 함 |
| 일부만 못 받음 | 그 사용자 `email` 필드가 비어 있음 — 이 경우 **에러 로그도 안 남고 조용히 건너뜁니다** |
| 보냈다는데 스팸함 | SPF `-all` — M365 경유가 아닌 경로로 나갔을 가능성 |
