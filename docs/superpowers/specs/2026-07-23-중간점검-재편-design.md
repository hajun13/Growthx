# 중간점검 재편 — 본부장·그룹대표 2단계 검토 + 이메일 + 변경 이력

- 날짜: 2026-07-23
- 상태: 설계 확정 (구현 계획 수립 전)
- 관련 모듈: `apps/api/src/modules/midterm`, `apps/web/src/features/eval-midterm`

## 1. 배경과 문제

현행 중간점검은 **전원 자가점검 방식**이다. 모든 피평가자가 KPI별 자가점검(`MidtermKpiCheckIn` — 상반기 실적·코멘트·자가등급)을 제출하고, KPI 결재선과 동일한 순차 확인 체인(1차 팀장 → 2차 본부장 → 최종 그룹대표)이 확인·반송한다.

실제 운영에서 필요한 것은 다르다.

- 임직원·팀장은 중간점검을 **수행하지 않는다**(자가점검 입력 부담 없음). 팀장은 평가자로도 참여하지 않는다.
- **본부장이 임직원의 KPI를 보고 코멘트**하고, 그 내용이 **이메일로 당사자에게** 전달되어야 한다.
- 임직원은 코멘트를 받아 **KPI 목표를 수정**하고, **그룹대표가 2차 검토**로 마무리한다.
- 전 과정이 **누가·언제·무엇을 어떻게 고쳤는지** 이력으로 남아야 한다.

즉 중간점검의 성격이 "자가 진척 보고"에서 **"상급자 주도 목표 재조정"** 으로 바뀐다.

## 2. 범위

**포함**
- 중간점검 참여자 구조·상태 머신 재설계
- 임직원의 KPI 목표 수정(기존 재조정 엔진 재사용)
- 4개 시점 이메일 발송
- 단계 타임라인 + KPI 필드 전/후 값 이력

**제외**
- 최종평가·연봉 산정 로직 (중간점검은 기존과 동일하게 **비구속** — 등급·보상에 반영되지 않음)
- 역량평가
- 기존 보완조치(`ActionItem`)·재조정 요청(`RebaselineRequest`) 화면의 사용자 흐름 변경 (내부 엔진만 재사용)
- SMTP 서버 구축 (§11 운영 의존성 참조)

## 3. 참여자와 검토 체인

### 3.1 대상자

해당 주기에서 다음을 모두 만족하는 사용자.

- 재직 중(`isActive = true`)
- 해당 주기에 **확정(`confirmed`) KPI 보유**
- 본인이 소속 그룹의 그룹대표(`Department.headUserId`, `type = group`)가 **아님**
- 본인이 소속 본부의 본부장이 **아님**

결과적으로 대상 = **임직원 + 팀장**. 본부장·그룹대표는 대상에서 제외된다.

### 3.2 체인 해석기 — `resolveMidtermReviewers`

기존 `resolveDownwardEvaluators`(`apps/api/src/common/access/access.util.ts:190`)와 **분리된 신규 함수**를 만든다. 기존 함수는 최종평가(3단계 0.5/0.3/0.2 가중)용이며 팀장을 1차로 포함하므로 그대로 쓸 수 없다.

부서 트리를 위로 올라가며 다음을 수집한다(기존 함수와 동일하게 `Department.headUserId` **명시 지정만** 사용, 권한 role로 추론하지 않음. 본인·비활성자는 제외).

| 자리 | 판정 |
|---|---|
| `firstReviewer` (1차) | 가장 가까운 상위 **본부장**(`type = division`의 `headUserId`) |
| — 본부장이 없으면 | 그룹의 **부그룹장**(`deputyHeadUserId`) |
| — 그것도 없으면 | 그룹대표 (단독 수행, 단계 표기 `1/1`) |
| `finalReviewer` (2차) | 소속 **그룹대표**(`type = group`의 `headUserId`) |

**엣지 케이스**

- `firstReviewer === finalReviewer` (그룹 직속 인원 등): 단계는 1개로 압축된다. 그룹대표가 코멘트 → 임직원 수정 → 그룹대표 승인. 화면 단계 표기는 `1/1`.
- `finalReviewer`가 없음(그룹 미지정 등): 대상자 생성에서 제외하고 HR 화면에 **경고로 노출**한다(조용히 누락하지 않음).
- 팀장은 어떤 경우에도 평가자 자리에 들어가지 않는다.

## 4. 상태 머신

`MidtermReviewStatus` enum에 **`commented`·`revised`·`returned`·`closed` 4개 값을 추가**한다. 미시작은 기존 `pending`을 그대로 재사용한다. 나머지 기존 값(`self_done`·`confirmed`·`revision_requested`·`rejected`)은 2025 등 과거 데이터 보존용으로만 남기고 신규 흐름에서는 생성하지 않는다.

```
pending   ──(1차 코멘트 등록)────────> commented    [메일① → 임직원]
commented ──(임직원 수정 제출)───────> revised      [메일② → 2차 검토자]
revised   ──(2차 반려)──────────────> returned     [메일③ → 임직원]
returned  ──(임직원 재수정 제출)─────> revised      [메일② → 2차 검토자]
revised   ──(2차 승인)──────────────> closed       [메일④ → 임직원·1차 평가자]
closed    ──(HR 되돌림)─────────────> revised
```

전이는 기존 `common/state/transitions.ts`의 `assertTransition`에 표를 추가해 강제한다.

**규칙**

- 각 단계의 액션은 **자기 차례일 때만** 수행 가능(기존 KPI 결재선·중간점검 확인의 순차 결재 원칙과 동일). 차례가 아니면 403 + 현재 대기 단계·담당자 안내.
- `hr_admin`은 대리 수행 가능하되, 대리 수행 시 이력에 **대리 표시**를 남긴다.
- `closed` 되돌림(`reopen`)은 **HR 전용**.
- 주기가 `closed` 상태이면 모든 쓰기를 차단한다(기존 사이클 쓰기 게이트와 동일).
- 임직원 제출 시 **KPI 변경 0건 + 사유만 작성**도 정상 제출로 인정한다("코멘트는 수용하되 목표 조정은 불필요" 케이스).

## 5. 데이터 모델

### 5.1 `MidtermReview` 변경

기존 자가점검용 필드(`selfNote`·`selfSubmittedAt`·`reviewStage`·`reviewTrail`)는 **과거 데이터 보존용으로 유지**하고 신규 흐름에서는 쓰지 않는다. 아래 필드를 추가한다.

| 필드 | 타입 | 설명 |
|---|---|---|
| `firstReviewerId` | `String?` | 1차 평가자(생성 시 스냅샷) |
| `firstComment` | `String?` | 1차 전체 총평 |
| `firstCommentedAt` | `DateTime?` | 1차 코멘트 등록 시각 |
| `memberNote` | `String?` | 임직원 회신 사유(수정 사유 / 미수정 사유) |
| `memberSubmittedAt` | `DateTime?` | 최근 수정 제출 시각 |
| `finalReviewerId` | `String?` | 2차 검토자(생성 시 스냅샷) |
| `finalComment` | `String?` | 2차 승인·반려 코멘트 |
| `decidedAt` | `DateTime?` | 2차 최종 판정 시각 |
| `revisionRound` | `Int @default(0)` | 임직원 수정 회차(반려 시 증가) |
| `singleStage` | `Boolean @default(false)` | 1차·2차 동일인(압축) 여부 — 화면 `1/1` 표기용 |

### 5.2 `MidtermKpiCheckIn` 재사용

신규 흐름에서 사용하는 필드는 다음과 같다. `self*` 필드는 신규 흐름에서 기록하지 않는다.

- `reviewerNote` — **1차 평가자의 KPI별 코멘트**
- `reviewerDecision` — 기존 값 체계를 그대로 사용: `'accepted'`(현행 유지) / `'rebaseline'`(조정 필요). 임직원 화면에서 어떤 KPI를 봐야 하는지 강조하는 데 사용
- `reviewerGrade` — 미사용(과거 데이터 보존)

### 5.3 `MidtermTrail` (신규)

JSON 배열 대신 정규 테이블로 만든다. 필터·정렬·감사 조회에 유리하고, 파일·모델 책임이 분리된다.

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | `String` | PK |
| `midtermReviewId` | `String` | FK (cascade) |
| `seq` | `Int` | 리뷰 내 순번 |
| `action` | `String` | `commented` / `revised` / `returned` / `approved` / `reopened` |
| `actorId` | `String` | 행위자 |
| `actorName` | `String` | **당시** 이름 스냅샷 |
| `actorPosition` | `String?` | **당시** 직책 스냅샷 (인사이동 후에도 "그때 누구였는지" 보존) |
| `onBehalfOf` | `Boolean @default(false)` | HR 대리 수행 표시 |
| `comment` | `String?` | 해당 단계 코멘트(총평) |
| `kpiChanges` | `Json?` | `[{ kpiId, kpiTitle, field, before, after }]` |
| `snapshotId` | `String?` | 수정 직전 `KpiSnapshot` id (전체 복원용) |
| `createdAt` | `DateTime` | 발생 시각 |

인덱스: `[midtermReviewId, seq]`, `[actorId]`. 스키마: `midterm`.

### 5.4 KPI 변경 적용

기존 재조정 엔진을 그대로 재사용한다.

- `validateProposal` — 수정 가능 필드는 **목표서술(`targetText`)·목표수치(`targetValue`)·가중치(`weight`)**. 가중치 합계 100% 검증 포함.
- `applyToKpis` — 실제 KPI 반영.
- `KpiSnapshot` — 수정 **직전** 상태를 `label = "중간점검 수정 전 (n차)"` 로 캡처.

적용 시점은 **임직원 제출 시점**이다(2차 승인까지 기다리지 않는다). 이유: 2차 검토자는 "수정된 결과물"을 보고 판정해야 하고, 반려되면 다시 수정 제출로 되돌아오므로 최종 상태는 항상 마지막 제출본과 일치한다. 각 회차의 직전 상태가 스냅샷으로 남아 되돌리기가 가능하다.

## 6. API

경로 접두사는 기존 `/midterm`을 따른다. 응답 봉투·에러 코드는 `api-contract-convention.md` 규약을 따른다.

| 메서드·경로 | 권한 | 설명 |
|---|---|---|
| `POST /midterm/reviews/open` | `hr_admin` | 대상자 일괄 생성(멱등). 1차 평가자에게 시작 안내 메일 |
| `GET /midterm/reviews` | 전체(역할별 스코프) | 목록. 피평가자=본인 건, 1차·2차=담당 건, HR=전체 |
| `GET /midterm/reviews/:id` | 본인·1차·2차·HR | 상세(코멘트·KPI·이력 포함) |
| `POST /midterm/reviews/:id/comment` | 1차 평가자(+HR 대리) | KPI별 코멘트 + 총평 등록 → `commented` |
| `PUT /midterm/reviews/:id/revision` | 피평가자 본인 | 수정안 임시저장(제출 아님) |
| `POST /midterm/reviews/:id/revision/submit` | 피평가자 본인 | 수정 확정 제출 → `revised` (KPI 반영 + 스냅샷) |
| `POST /midterm/reviews/:id/approve` | 2차 검토자(+HR 대리) | 승인 → `closed` |
| `POST /midterm/reviews/:id/return` | 2차 검토자(+HR 대리) | 반려(사유 필수) → `returned` |
| `POST /midterm/reviews/:id/reopen` | `hr_admin` | 확정 되돌림 → `revised` |
| `GET /midterm/reviews/:id/trail` | 본인·1차·2차·HR | 이력 타임라인 |

기존 `PATCH /midterm/reviews/:id/confirm`·`request-revision`·`reject`는 과거 데이터 조회 호환을 위해 남기되, 신규 상태값에 대해서는 400을 반환한다.

OpenAPI 발행 후 `packages/contracts` orval 재생성이 필요하다.

## 7. 화면

### 7.1 1차 평가자(본부장) — `/eval/midterm`

기존 부서장 평가 화면과 동일한 **master-detail** 구조를 재사용한다(`EvaluationSubjectPanel` 패턴: 좌측 대상자 목록 + 검색 + 상태 필터 + 제출 후 다음 미착수 대상 자동 이동).

- 우측: 대상자의 확정 KPI 목록. 각 KPI에 **코멘트 입력 + 판정(현행 유지 / 조정 필요)**
- 하단: 전체 총평 + 제출(`sticky` 액션 패널)
- 제출 전 확인 모달("코멘트가 대상자에게 이메일로 발송됩니다")

### 7.2 임직원 — `/eval/midterm`

- 상단: 본부장 코멘트(총평) 배너
- KPI 목록: 코멘트가 달린 KPI 강조. 목표서술·목표수치·가중치 인라인 편집
- 가중치 합계 실시간 표시(100% 미달·초과 시 제출 차단)
- 하단: 회신 사유 입력 + 제출. **변경 0건일 때는 사유 필수**
- 미저장 이탈 가드(`beforeunload`) — 기존 5개 화면과 동일 패턴

### 7.3 2차 검토자(그룹대표)

1차와 같은 master-detail. KPI별로 **변경 전/후가 나란히** 표시되고(변경분 하이라이트), 하단에 승인 / 반려(사유 필수).

### 7.4 이력 타임라인

상세 하단 카드. 예시:

```
● 본부장 민명기  1차 코멘트           7/24 14:20
  "상반기 수주 진척이 목표 대비 낮습니다. 매출 목표 현실화 검토 바랍니다."
  KPI 3건 지적

● 홍길동  수정 제출 (1회차)            7/25 09:10
  매출액 목표  50억 → 40억
  가중치       30% → 25%
  사유: "상반기 수주 지연 반영"

● 그룹대표 홍두화  반려               7/25 16:40
  "가중치 조정 사유가 불충분합니다."
```

### 7.5 HR

평가 운영 화면에 **"중간점검 개시"** 버튼과 진행 현황(단계별 인원수, 미착수자, 1차 평가자 미지정 경고).

## 8. 이메일

`NotificationsService.notifyUser`를 중간점검 모듈에 주입해 **인앱 알림 + 이메일**을 동시에 발송한다(현재 중간점검 모듈은 알림 서비스를 주입하지 않아 메일이 나가지 않는다).

| 시점 | 수신자 | 알림 타입 |
|---|---|---|
| ① 1차 코멘트 등록 | 피평가자 | `midterm_comment_received` |
| ② 임직원 수정 제출 | 2차 검토자 | `midterm_revision_submitted` |
| ③ 2차 반려 | 피평가자 | `midterm_returned` |
| ④ 2차 승인(마감) | 피평가자 + 1차 평가자 | `midterm_closed` |
| (부가) 개시 | 1차 평가자 전원 | `midterm_opened` |

**본문 형식: 요약 + 링크.** 평가 코멘트 전문은 메일에 싣지 않는다(사내 메일함·백업에 평가 정보가 잔존하는 위험 회피).

```
제목: [에너지엑스 인사 평가] 중간점검 코멘트가 등록되었습니다

홍길동 님,
민명기 본부장님이 2026년 중간점검 코멘트를 남겼습니다.
· 조정 검토 요청 KPI: 3건
· 회신 마감: 2026-07-31

내용 확인 및 수정 → https://.../eval/midterm
```

발송 실패는 비즈니스 트랜잭션을 막지 않는다(기존 `MailService` 베스트 에포트 정책 유지). 마감 리마인더는 기존 일정 기반 D-N 자동 발송(`runDueReminders`)을 `mid_review` 단계에 연결해 재사용한다.

## 9. 이력·감사

- 모든 상태 전이에서 `MidtermTrail` 1행 + `AuditService.record` 1건을 **같은 트랜잭션 안에서** 기록한다(감사 로그 실패는 기존 정책대로 트랜잭션을 막지 않는다).
- 열람 권한: **본인 + 1차 평가자 + 2차 검토자 + HR**. 그 외 403.
- HR 감사 화면에서 `entity = MidtermReview` 필터로 교차 조회 가능.

## 10. 마이그레이션·기존 데이터

1. 마이그레이션: enum 값 추가, `MidtermReview` 필드 추가, `MidtermTrail` 테이블 생성. 기존 행은 손대지 않는다(모든 신규 필드 nullable / default).
2. 2025 등 과거 주기의 자가점검 데이터·화면은 **읽기전용**으로 보존한다. 과거 주기 상세는 기존 렌더링 경로를 그대로 탄다.
3. 2026 주기에서 자가점검 입력 UI는 노출하지 않는다.
4. 현재 2026 주기에 남아 있는 진행 중 자가점검 행이 있으면, `POST /open` 실행 시 **신규 흐름으로 초기화**한다(기존 `selfNote`·`reviewTrail`은 보존, 상태만 `pending`으로). 초기화 대상은 실행 전 미리보기로 HR에게 보여준다.

## 11. 운영 의존성 — SMTP (선행 필요)

현재 `.env`에 SMTP 설정이 없다(`.env.example`에도 주석 처리). 이 상태에서는 `MailService`가 콘솔 로그로 폴백하므로 **메일이 실제로 발송되지 않는다.**

필요 항목: `SMTP_HOST`, `SMTP_PORT`, `SMTP_SECURE`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM`.

구현은 SMTP 유무와 무관하게 진행하고(콘솔 폴백으로 검증 가능), 실제 운영 발송은 값 확보 후 `.env` 주입 + 발송 스모크 테스트로 마무리한다.

## 13. 검증 계획

**단위(vitest)**
- `resolveMidtermReviewers`: 팀 소속 임직원 / 팀장 / 본부 직속 / 본부 없는 그룹 직속(부그룹장 대체) / 부그룹장도 없음(대표 단독) / 그룹대표 본인(대상 제외) / 본부장 본인(대상 제외)

- 가중치 합계 검증: 100% 미달·초과 차단, 정확히 100% 통과, 변경 0건 통과

**통합(라이브 스택)**
- 전체 시나리오 1회: 개시 → 1차 코멘트 → 메일 로그 확인 → 임직원 수정 제출 → 2차 반려 → 재수정 → 승인 → `closed`

- 권한: 팀장이 평가자 목록에 나타나지 않음, 타인 이력 403
